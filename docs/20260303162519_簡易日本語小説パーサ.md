# 簡易日本語小説パーサ

カクヨムのような簡易記法によるHTML変換パーサを実装したい。

但し一つのファイルでメタデータや全章を含めたものであり10万字以上を想定する。作品を完結できる全内容を包含するため数百万字規模も想定する。

フリーズせず迅速にページ表示したい。スクロールではなく本のようにページ遷移したい。画面サイズに応じて読みやすいフォントサイズに自動調整したい。URLにより指定箇所を迅速に表示したい。

# ブロック要素

`<em>`, `<ruby>`はインライン要素だった。これに対して次はブロック要素を考える。

ブロック要素は文書の冒頭と末尾以外は前後に二連続改行を入れることで分割するものとする。

ファイルの冒頭にはフロントマターとして文書のメタデータをセットできる。

```markdown
タイトル：　副題
https://...
https://...
キャッチコピー：　サブキャッチ（なんちゃら賞受賞！等）
＠著者名
https://...
https://...
yyyy-mm-dd〜yyyy-mm-dd（発案〜執筆〜校正）
yyyy-mm-dd〜yyyy-mm-dd（公開〜最終更新日）
カテゴリ：小説＞
タグ：

＃見出し１

一つ目のパラグラフ。
形式段落《けいしきだんらく》。

＊二つ目＊のパラグラフ。

＃＃見出《みだ》し２

パラグラフ。
```

## フロントマター

```markdown
タイトル：　副題
https://...
https://...
キャッチコピー：　サブキャッチ（書籍化、コミカライズ、なんちゃら賞受賞！等）
https://...
https://...
＠著者名
https://...
https://...
yyyy-mm-ddT00:00:00〜yyyy-mm-ddT00:00:00（発案〜執筆〜校正）
yyyy-mm-ddT00:00:00〜yyyy-mm-ddT00:00:00（公開〜最終更新日）
分類（カテゴリ、ジャンル）：小説＞ラノベ＞ファンタジー
語札（タグ）：最強、終末、令嬢、チーレム等
警告（セルフレイティング）：性、暴力、残酷
免許（ライセンス）：©、CC0、CC-BY、CC-BY-SA、CC-BY-ND、CC-BY-NC、CC-BY-NC-SA、CC-BY-NC-ND
```

一行目はタイトル（必須）。`：　`が含まれていれば、その前が主題、その後ろが副題。
タイトルの下行に`https://`から始まる値があれば、この文書を表示するHTMLサイトや、原稿テキストファイルへのリンクやGitHubリポジトリなど関係するURLとする。

タイトルの次はキャッチコピー（任意）。`：　`が含まれていれば、その前が主キャッチ、その後ろが副キャッチ。
キャッチの下行に`https://`から始まる値があれば、この文書に関する書籍販売URLや評価されたサイトへのリンクとする。

タイトルやキャッチコピーは、分類、語札、警告、免許と違って固定テキストをキーにしておらず位置がキーであることに注意。

行頭`＠`なら著者名（必須）。その下行に`https://`から始まる値があれば、著者のサイトやSNSなど著者に関連するサイトへのリンクとする。

`yyyy-mm-ddT00:00:00`は日時（任意）。一行目の日時は公開前まで。二行目の日時は公開後から最終更新まで。

ここまでの項目は位置（順序）が固定であり位置によってその内容を特定する仕組みである。

分類、語札、警告、免許は任意。特定のキーワードをキーにして指定するため、位置は日付より下であれば順不同。

分類はカテゴリとジャンルを包含した概念。木構造で包含関係を`＞`か`＜`で示す。どちらの方向から始めても構わないが、必ずどちらか一方向のみ。`カテゴリ`や`ジャンル`に分けて書いても良い。カテゴリは文書としての分類で、ジャンルは内容の傾向（カテゴリより詳細）。

語札はタグのこと。どのような属性の人が好んで読むか。その属性を表す語。嫌いなものを避けるために使われることもある。流行に支配されており流行を理解していないと読み取れず書けない。

警告はセルフレイティングのこと。読者がダメージを受けぬよう最低限の警告を行う。

免許は文書のライセンスのこと。再配布、改変（二次創作の公開）の自由度を示す。もし免許の項目がなければ無条件で`©`であり通常通り著作権がある。もし免許の項目があればライセンス名を表記する。基本的にCC（クリエイティブ・コモンズ・ライセンス）が対象。通常の著作権を意味する`©`も含めれば8種類。それ以外に小説専用のライセンスはまだ世界に存在しないと思われる。たとえば人物や世界観など設定だけは絶対に守れ等。しかし抽象度が高く、そのための設定集を必要とし、作者の独断と偏見でしか判断できず実現は難しい。

## 見出し

```markdown
＃見出し１
＃＃見出し２
＃＃＃見出し３
＃＃＃＃見出し４
＃＃＃＃＃見出し５
＃＃＃＃＃＃見出し６
```

見出しは行頭に`＃`が`1`〜`6`個連続している時に該当する。

見出し文字の一文字目に`＃`を使いたい場合もあるかもしれない。この時は見出しメタ文字との間に全角スペースを入れる。以下のように。尚、見出し文字の先頭と末尾に全角スペースは使えない。トリムされるものとする。

```markdown
＃＃＃　＃見出し３
```

以下の書式も考えたが、所々の理由で却下した。見出しの最初に全角数字を使いたい場合は全角スペースを挟めば良い。

```markdown
＃タイトル
＃１見出し１
＃２見出し２
＃３見出し３
＃４見出し４
＃５見出し５
＃６見出し６
```
```markdown
＃３　３見出し３
```

この番号は深度ではなく順序に見えてしまうため紛らわしい。ページ番号のような順番に見えてしまう。よって数による`＃`略記法の採用は見送り`＃＃＃見出し３`のように`＃`を複数個書く表記を採用した。

### 自動採番

実際、順序についてはどうするか。見出しが確定した時点で自動採番するのが良いだろう。

となると原稿テキストでハードコーディングする訳にはいかない。また、設定によって採番アルゴリズムを変えたい。すなわち自動採番するのか、するなら開始は`0`か`1`かをユーザに設定させたい。

しかし間に`間話`があり、採番して欲しくない章がある場合も考えられる。`序章`,`間章`,`終章`,`外伝`など`１章`のように数値でないものを差し込みたい場合もあるし、数値にしたって小文字、大文字、漢数字（大字、小字）のパターンもある。あるいは丸囲み文字を使いたいかもしれない。`2章1節　山田の冒険⑦`のように複数の数が前後にある場合も考えられる。

中々に面倒である。自動化できることかも疑問だ。`{chap}{sec}山田の冒険{pg}`のように番号をプレースホルダーで表記する案も考えられる。

### slug

内部リンク用ID。これは自動採番できればそれをIDにできるだろう。

```html
<a href="#1-1">見出し1-1</a>

<h1 id="1-1">見出し1-1</h1>
```

但しこれが必要なのは同一文書ファイルに複数の見出しがある時だけだ。見出し毎に別ファイル化したり、DOMにより一つの見出し範囲ずつしか出力しないようにしていれば不要だ。しかしURLで直接指定することで、所定の見出し範囲の文書を取得できるようにしたいため、どのみちIDは欲しい。

目次（見出し一覧）にも使えるだろう。

### 副題

見出しには副題が欲しい時がある。以下のようにして書く。

```markdown
＃主題
副題
```
```html
<hgroup>
  <h1>主題</h1>
  <p>副題</p>
</hgroup>
```

しかし主題と副題の間を改行したい場合もあれば、`主題〜副題〜`のように一行で表示したい場合もあるだろう。これをどう調整するか。以下の様に書式を指定させるか？

```
hgroup: {main}<br>{sub}
```
```
hgroup: {main}〜{sub}〜
```

```html
<hgroup data-form="{main}<br>{sub}">
  <h1>主題</h1>
  <p>副題</p>
</hgroup>
```

問題はその書式をどう簡易記法で書かせるかだ。HTMLを直接書く書式すら許容するので雑な書式ではダメだ。以下のようにするか？

```markdown
---setting
hgroup:
	form: {main}<br>{sub}
---
```

これまで全角文字だけで雑にメタデータを埋め込んでいたが、詳細設定に関しては半角文字で詳細に入力することになる。しかしこの書式は、ブロック、インラインとは別の「フェンス」という概念を取り入れる必要が出てくる。一気に実装難易度が高まりそうだから、今回は保留にする。

今回は`{main}<br>{sub}`という書式で固定にする。というか基本的には改行しないほうに特殊な実装をせねばならないため、以下のようにすれば主題と副題は改行されて表示されるはず。

```html
<hgroup>
  <h1>主題</h1>
  <p>副題</p>
</hgroup>
```

## パラグラフ

二連続改行の中にある一つの改行は`<br>`になる。

### 会話文

`＜`をメタ文字に使う。

小説には地の文と会話文の二種類ある。他にも引用文や遠方からの電話など異なる文脈があるが、いずれにせよ地の文と会話文は基本的に鉤括弧で区別している。この時、セリフの記法を用意すると便利か。

```markdown
＜ここにセリフを入れると左右を鉤括弧で閉じ、末尾の句点も消す。セリフとしての禁則事項とか決まりごとをやってくれる。改行は含められない。
```

```markdown
＜
ここにセリフを入れると複数行のセリフになる。

改行を含めることができる。

長い文章をセリフにするのは小説として微妙だが、かといって地の文にすると繰り返しになりくどいし、無理やり一行のセリフにすると長すぎて読みづらい。そんな場面で使う。
＜
```

HTML的には地の文と区別できる要素はない。`＜`を使わずにテキストで`「」`を使えば見分けが付かなくなってしまうだろう。

それでも会話文に`＜`を使うようにすれば、以下のように会話文の表示方法を切り替えることができるようになるかもしれない。

* 鉤括弧で会話文を示す
* CSSの吹き出しで会話文を示す
* 吹き出しをクリックして次の会話文を出す（これは無理か）

小説という文脈においては鉤括弧で会話文を示すべき。

懸念事項は、原稿では会話文が`「」`の書式ではないことか。わざわざ`＜`書式にする必要があるか疑問である。そのため実装優先度は低い。

## 場面転換

小説には以下のように場面転換や視点転換を読者に知らせるメタテキストがある。

用途|例
----|--
場面転換|`＊`,`◆◇◆◇`
視点|`山田視点――`, `とある一室――`, `時は西暦一九九Ｘ年――`
完了|`完`, `FIN`, `終劇`, `第一部　完`

これらを作品の本文にするには微妙だが、かといって書かないと読みにくい。しかし丁寧に説明すると冗長になるため、こうした略記に頼りたい場面がある。作品本文として文字数カウントなどはされたくないが、文字表示はしたい時がある。

HTMLでは`<hr>`が近い。セマンティクスとしては`Thematic Break`(話題転換)だ。

```html
<hr>
<hr class="scene" data-text="◆◇◆◇" data-align="center">
<hr class="view" data-text="山田視点">
<hr class="fin" data-text="完" data-align="right">
```
```markdonw
＊＊＊
```
```markdonw
＊＊＊｜◆◇◆◇
```
```markdonw
＊＊＊＞完
```
```markdonw
＊＊＊＜第一部　完
```
```markdonw
＊＊＊｜山田視点
```

行頭`＊＊＊`で`<hr>`に変換する。それはデフォルト設定だ。`data-text`はユーザ指定値であり無ければ`◆◇◆◇`。配置はユーザ指定であり無ければ`left`。その後ろに`＜`か`｜`か`＞`があれば配置は`left`,`center`,`right`。それらが無ければ`left`。もし`＜`,`｜`,`＞`をテキストの一文字目に使いたければ`＊＊＊　｜をメタ文字にせず一文字目に使う`のように`＊＊＊`の直後に全角スペースを入れること。

```markdonw
＊＊＊　｜、＜、＞をメタ文字にせず一文字目のテキストとして使うにはアスタリスク三連続の直後に全角スペースを入れること
```

もしフェンス記法を実装したらユーザはよく使うパターンを以下のように指定できるようにしたい。

```markdonw
---setting
thematic-break:
	default:
		text: ◆◇◆◇
		align: left	
---
```

## まえがき／あとがき

作品ではなく著者が読者に直接話しかけるメタい文脈。前回のあらすじや全然関係ない話など。作品とは関係ないが執筆に影響している話など。

読みたい人もいれば、読みたくない人もいる。本文以上に宣伝している人までいる。

一つのパラグラフで済むなら以下。メタ文字は`＠`で`まえがき`は任意テキスト。

```html
＠まえがき
三行で
前回のあらすじを
書いたりする
```

複数のパラグラフが必要なら以下。

```html
＠＠＠まえがき
著者が読者に直接話しかけるメタい文脈。

複数ブロックもOK。インラインもOK。漢字《かんじ》。

見出しはNG。大規模に書くのはダメ。
＠＠＠
```

この著者メタ文脈への対処は次の通り。

* 全文テキスト表示する(`<fieldset><legend>`のような別枠である表示をする。この概念をboradとする)
* 全文削除する（作品外は見たくない）
* 全文`<detail><summary>まえがき</summary>本文</details>`表示する（``クリックしないと本文が表示されない）

これは読者の設定次第で表示を動的変更することになるか。設定の実装は大変そうなので現状は全文テキスト表示で固定にする。

```css
section.board {
    border: 1px solid currentColor;
    border-radius: 0.5em;
}
```
```html
<section class="board">
  <h2>まえがき</h2>
  <p>本文１</p>
  <p>本文２</p>
</section>
```

# 1. 現状確認

# `<ruby>`, `<em>`

Markdownのように簡易記法で`<ruby>`, `<em>`を書けるようにしたい。

* `<ruby>`: ルビ。日本語では振り仮名。
* `<em>`: 強調。日本語では傍点(圏点)。含みを持つ。`<b>`や`<strong>`とは異なり重要さを意味した強調ではなく含みを持っていることを示す。英語圏では斜体`<i>`だが、日本語では傍点(圏点)で示すのが一般的。

## `<ruby>`

`<ruby>`は以下四要素の組合せパターンがある。

* 部品: `<rb>`, `<rt>`, `<rp>`
* 用法: 振り仮名、義訓
* 位置: 上、下、両
* 分割: モノ、熟語、グループ
* 読上: アクセシビリティ

### 部品

ルビ`<ruby>`には内部に含める部品がある。

部品|名前
----|----
`<rb>`|`Ruby Base`|親文字
`<rt>`|`Ruby Text`|ルビ文字
`<rp>`|`Ruby Parenthesis`|ルビ括弧

公式仕様[HTML Living Standard][]では既に`<rb>`は削除されており`<ruby>`にテキストノードとしてそのまま入力することになっている。しかし依然として親文字という概念は健在する。

[HTML Living Standard]:https://html.spec.whatwg.org/multipage/

ルビは親文字に対してルビ文字を付与する。ルビ括弧は、ルビ機能に非対応のブラウザで用いるフォールバック用の要素である。

```html
<ruby>超電磁砲<rp>（</rp><rt>ちょうでんじほう</rt><rp>）</rp></ruby>
```

### 用法

ルビ（和訳:`振り仮名`）には以下二つの用法がある。

用法|概要
----|----
読み仮名|漢字の正しい読み方（音読み・訓読み）を示す。
義訓|本文の漢字とは別の、意味やイメージに合わせたカタカナや言葉を読みとして当てる。

```html
<ruby>超電磁砲<rt>ちょうでんじほう</rt></ruby>
```
```html
<ruby>超電磁砲<rt>レールガン</rt></ruby>
```

どの用途であるか機械判定させるために`data`属性値を使うことも検討できる。

```html
<ruby>超電磁砲<rt data-yomi>ちょうでんじほう</rt></ruby>
```
```html
<ruby>超電磁砲<rt data-gikun>レールガン</rt></ruby>
```

### 位置

ルビの表示位置。横書きなら上、下、両方の三通り。縦書きなら右、左、両方の三通りある。デフォルトは開始位置(上/右)である。

書字方向|ルビ位置
---------|--------
横書き|上、下、両方
縦書き|右、左、両方
一般的|始、終、両方

CSSで以下のように実装する。

```css
ruby.over {ruby-position:over;}
ruby.under {ruby-position:under;}
```

HTMLでCSSを利用すると以下。

```html
<ruby class="over">超電磁砲<rt>レールガン</rt></ruby>
```
```html
<ruby class="under">超電磁砲<rt>レールガン</rt></ruby>
```

#### 両面ルビ

両方に付与するときは`<ruby>`をネストし、位置を調整する。

```html
<ruby class="over"><ruby class="under">超電磁砲<rt data-gikun>レールガン</rt></ruby><rt data-yomi>ちょうでんじほう</rt></ruby>
```
```html
<ruby class="under"><ruby class="over">超電磁砲<rt data-gikun>レールガン</rt></ruby><rt data-yomi>ちょうでんじほう</rt></ruby>
```

### 分割

ルビは分割方法によって記法が変わる。

分割|概要
----|----
モノ|親文字一字に対してルビを振る。（親文字は漢字）
熟語|熟語全体に対してルビを振る。（親文字は全て漢字）
グループ|非漢字を許容する全親文字に対してルビを振る。

分割|例
----|--
モノ|`<ruby>学<rt>まな</rt></ruby>び`
熟語|`<ruby>学校<rt>がっこう</rt></ruby>`
グループ|`<ruby>伸縮自在の愛<rt>バンジーガム</rt></ruby>`

例えば同じ`山田太郎`でも、以下のように書ける。

分割|例
----|--
モノ|`<ruby>山<rt>やま</rt>田<rt>だ</rt>太<rt>た</rt>郎<rt>ろう</rt></ruby>`
熟語|`<ruby>山田<rt>やまだ</rt>太郎<rt>たろう</rt></ruby>`
グループ|`<ruby>山田太郎<rt>やまだたろう</rt></ruby>`

モノは漢字として最も正しい。熟語は氏名を分離している。グループは氏名全部含めている。

別にどんな分割でも良いのでは？　と思うだろうが、致命的な問題がある。

#### テキスト検索

じつはテキスト検索する時は要素単位で分割されてしまう。

たとえば`山田太郎`をモノルビで書いたら以下になるが、この時これをブラウザで表示しつつ`やまだ`と検索してもヒットしない。ブラウザの種別やバージョンによっては`山田`でもヒットしない。

```html
<ruby>山<rt>やま</rt>田<rt>だ</rt>太<rt>た</rt>郎<rt>ろう</rt></ruby>
```

#### 氏名分割

日本語では人名を複数の漢字で表しつつ、更に氏と名に分かれる。この時、氏と名の単位で熟語ルビにしたい場合もあれば、グループルビにしたい場合もある。あるいはモノルビにしたい場合もある。いずれにせよ選択肢があるため同じ親文字とルビ文字であっても分割方法が変わり、異なるHTMLになる。

#### グループルビの多様性

グループルビはモノ・熟語ルビと違い、非漢字をも許容する。例えば以下。`の`という非漢字を親文字に含めることが可能だ。また、グループルビは正式な読み方ではなく義訓を許容する。その両方の組合せが以下だ。

```html
<ruby>伸縮自在の愛<rt>バンジーガム</rt></ruby>
```

* グループルビは親文字が必ずしも「漢字のみ」とは限らない（平仮名など非漢字を許容する）
* グループルビのルビ文字は必ずしも正式な漢字の読みだけとは限らない（義訓を許容する）

### 読上

アクセシビリティを考慮する。スクリーンリーダー上ではルビ文字が読み上げられる。しかし、場合によっては読み上げられたくない場合がある。読み上げると意味が分かりにくくなったりするような場合もある。この時、読み上げを抑制したい。

例えば両面ルビ。アクセシビリティ上の混乱が発生しかねない。そこで二つある`<rt>`のうち一つを`aria-hidden="true"`で無効化する。以下のように。

```html
<ruby class="over"><ruby class="under">超電磁砲<rt aria-hidden="true" data-gikun>レールガン</rt></ruby><rt data-yomi>ちょうでんじほう</rt></ruby>
```

## `<em>`

`<em>`は含みを持つ強調だ。日本語では傍点(圏点)である。

`<b>`や`<strong>`とは異なり重要さを意味した強調ではなく、含みを持っていることを示す。英語圏では斜体`<i>`で表示されるが、日本語では傍点(圏点)で示すのが一般的だ。

ごま`﹅`による`<em>`表示はCSSで以下のように示す。

```css
em {
  font-style: normal;
  text-emphasis: filled sesame;
  -webkit-text-emphasis: filled sesame;
}
```

それをHTMLで使うと以下。

```html
「お姉様、<em>あれ</em>を使うわ」

「ええ、良くってよ」
```

## `<ruby>`と`<em>`の併用

基本的に`<ruby>`と`<em>`の併用は物理的に不可能だ。但し、Firefox 142で動作確認した所、以下のようなHTMLでルビの上に傍点の表示を確認した。

```html
<ruby><em>超電磁砲</em><rt>ちょうでんじほう</rt></ruby>
```
```html
<ruby>超電磁砲<rt><em>ちょうでんじほう</em></rt></ruby>
```
```html
<em><ruby>超電磁砲<rt>ちょうでんじほう</rt></ruby></em>
```

どうやら`<ruby>`と`<em>`の親子関係を問わず、必ず傍点がルビの上に表示されるようだ。但し`<rt>`内に`<em>`があると傍点の数は`<rt>`内字数になるし、それ以外であれば親文字の字数になる。また傍点の大きさも`<rt>`内だと小さくなるようだ。

ルビの上に傍点があるなら、ルビの数と同じであるほうが自然だ。特に上記例の場合、親文字だと４字、ルビ文字だと８字になり、まったく異なる。但し、`<rt>`内`<em>`だと傍点が小さくなり強調度合いが少ない。かといってルビより少ない４字の傍点を８字のルビの上に配置するのも不自然である。親文字の直上ならば自然だったのに。

つまり`<ruby>`と`<em>`を併用する場合は、以下のような点に配慮すべきである。

* 傍点は強調という意味の特性上、最も早く目に入る所に配置すべきである
* 傍点は字に付随する特性上、同じ字数であるべきである

これを踏まえると、次のような解決策を思いつく。

* 傍点は上に配置する（デフォルト）。対象は親文字。
* ルビは下に配置する。

すると以下のようなHTMLになる。

```html
<ruby class="under"><em>超電磁砲</em><rt>ちょうでんじほう</rt></ruby>
```

一応、傍点とルビの位置を逆転することはできた。技術的にも表示的にもFirefox142で以下の方法で確認した。ポイントはCSSの`text-emphasis-position: under;`だ。

```css
text-emphasis-position: under;
```
```css
em.under {
  font-style: normal;
  text-emphasis: filled sesame;
  -webkit-text-emphasis: filled sesame;
  text-emphasis-position: under;
}
```
```html
<ruby class="over"><em class="under">超電磁砲</em><rt>ちょうでんじほう</rt></ruby>
```

上記で、ルビを上、傍点を下に配置できた。しかし傍点の強調という意味からすれば、先に視界に入る位置に配置すべきである。よって技術的に可能であっても傍点を下に配置する合理性は無い。

### ルビと傍点の同列化は禁止

ルビと傍点を同じ段に表示することも可能だ。ようはルビ文字の変わりに傍点文字を付与すれば良い。これなら一段分でルビと強調を併用できる。但し、それをする場面はレアケースであり、かつセマンティクスやアクセシビリティ上の問題があるため、実際には使わないほうが良いと思われる。

```html
<ruby>前後<rt>まえ</rt>強調<rt>﹅﹅</rt>後<rt>あと</rt></ruby>
```

但し、これはセマンティック・ウェブに反する。強調する意図としての`<em>`要素がないため、正しく意味付けできていない。

また、読み上げに関しては`aria-hidden="true"`を付与すべきだろう。この時、スクリーンリーダーがどうなるか不明だが「きょうちょう」と読み上げられなくなってしまうと思われる。

```html
<ruby>前後<rt>まえ</rt>強調<rt aria-hidden="true">﹅﹅</rt>後<rt>あと</rt></ruby>
```

## 用法・用量

`<ruby>`や`<em>`は乱用すると読みづらい文書になってしまう。適切な用法・用量は読者によって変わるため対象年齢に合わせた用量を概算しておく必要がある。

傍点の下配置は用法・用量としての合理性がない。それどころか傍点を下に配置することで、その行に要する幅が増えてしまい、全体的に余計な行間が増えてしまう。これは画面あたりの表示行数・字数を減らし、スクロール回数の増加につながってしまう。

ルビや傍点の追記は画面あたりの字数量を考慮する必要がある。乱用すれば読みづらいページ構成になってしまうだろう。

過剰な装飾は品質を落とす。良い文章は装飾でなく文章自体の構成や使う文字の秀逸さで決まる。

# 2. 簡易記法の考察

ここまで`<ruby>`と`<em>`の現状把握をした。これをもっと楽に書けるようにしたい。そのための簡易記法について考察する。

## 既存記法

既存では青空文庫、カクヨム、なろう、でんでん記法などがある。これらは規定のメタ文字を使うことで簡易記法を実現している。

今回の`<ruby>`と`<em>`が想定される場面は日本語の文章内である。つまり日本語入力を妨げないような記法にしたい。すると以下のような観点が重要になる。

* 日本語入力しやすいメタ文字を使う
* 文章内では使わない字をメタ文字にする

### メタ文字の候補

全角が望ましい。

#### 半角

半角文字。`|`,`{`,`}`等。画面占有量が少なく、半角という字幅によりメタ文字と自然言語との区別がしやすい。但し入力時は`半角／全角`を切り替えねばならず面倒。また、等幅フォントであっても半角と全角のサイズ差のせいで位置ずれが起きて原稿が読みづらくなる。

```html
|漢字{かん|じ}
```

#### 全角

全角文字。`｜`,`《`,`》`等。これらは等幅フォントであれば位置ずれが起きず綺麗に位置が整う。但しメタ文字の幅が大きいため画面占有量が半角よりも大きくなる。また、`《》`の入力は`半角／全角`せずとも良いものの、IMEによる入力変換が必要になる。`｛｝`のほうが簡単か。

```html
｜漢字《かんじ》
```
```html
｜漢字｛かんじ｝
```

#### 流用性

* 他の小説サイトでも流用できるよう同じ記法にすべきか
* それとも更に簡易記法を考え、それを各サイト形式に変換すべきか
* 相互変換できるようにすべきか

### `<em>`

```markdown
＊ここを強調＊する。
```
```html
<em>ここを強調</em>する。
```

### メタ文字

強調のメタ文字は`＊`だが以下の懸念点がある

* `＊`は場面転換や注釈でしばしば使われる。果たしてメタ文字にして良いものか（共用は可能だが少し紛らわしいかも？）
* かといって半角`*`にすると入力時に切替が面倒だし、Markdownの太字`<b>`と重複する
* カクヨムのように`《《強調》》`とすると冗長だ

## `<ruby>`

親文字が漢字で、かつ`《》`の中にルビ文字を入れる方法。

```html
漢字《かんじ》
```
```html
<ruby>漢字<rt>かんじ</rt></ruby>
```

親文字が非漢字でも良いよう親文字の開始位置をパイプ`｜`で指定する方法。

```html
｜ABC《えーびーしー》
```
```html
<ruby>ABC<rt>えーびーしー</rt></ruby>
```

しかし既存の記法では以下が不可能だ。

* 下ルビ
* 両面ルビ
* 分割ルビ
* em併用ルビ

### 下ルビ

ルビ文字の先頭に`｜`を付与する。

```markdown
漢字《｜した》
```
```html
<ruby class="under">漢字<rt>した</rt></ruby>
```

```markdown
｜KANJI《｜した》
```
```html
<ruby class="under">KANJI<rt>した</rt></ruby>
```

### 両面ルビ

```markdown
漢字《うえ｜した》
```
```html
<ruby><ruby class="under">漢字<rt>した</rt></ruby><rt>うえ</rt></ruby>
```

両面ルビの場合、以下のパターンが考えられる。これをどう制御するか。

* 下ルビのほうは読み上げられないようにする
* 上ルビのほうは読み上げられないようにする
* 両方読み上げる

基本的には「読み上げを考慮しない」ことにする。理由は以下。

* 両方読み上げられると冗長ではあるが、表記内容を全て視聴できる
* 読み上げを削除するコードを書くと冗長になるため削除する

文字としては見たいけど、読み上げはされたくない。そんな場合もあるはずだ。しかしそれには非対応とする。理由は「簡易記法」が簡易でなくなってしまいそうだから。

### 分割ルビ

グループルビではなく、分割ルビ（モノルビ、熟語ルビ）にする方法。分割ルビは複数の`<rt>`を作る。区切り位置には読点`、`を使う。以下の二通りある。

* モノルビ：親文字に読点がなく、ルビ文字に読点がある
* 熟語ルビ：親文字に読点があり、ルビ文字に読点がある

#### モノルビ

```markdown
山田太郎《やま、だ、た、ろう》
```
```html
<ruby>山<rt>やま</rt>田<rt>だ</rt>太<rt>た</rt>郎<rt>ろう</rt></ruby>
```

#### 熟語ルビ

```markdown
｜山田、太郎《やまだ、たろう》
```
```html
<ruby>山田<rt>やまだ</rt>太郎<rt>たろう</rt></ruby>
```

読点があるせいで先頭のパイプを必要としてしまう。パイプをなくせばルビ文脈でなく自然文章の読点と区別がつかなくなってしまう。故に熟語ルビは親文字がすべて漢字でもパイプを必要とする。

尚、以下のようにしてしまうと別々の`<ruby>`要素になってしまう。一人の人名を表すためには一つの`<ruby>`要素にしたい。よって熟語ルビを使う。

```markdown
山田《やまだ》太郎《たろう》
```
```html
<ruby>山田<rt>やまだ</rt></ruby><ruby>太郎<rt>たろう</rt></ruby>
```

### em併用ルビ

`<em>`単体は以下だ。

```markdown
＊ここを強調＊する。
```
```html
<em>ここを強調</em>する。
```

これを`<ruby>`と併用する。

* 傍点
	* 対象: 親文字／ルビ文字
	* 範囲: 開始〜終了

さて、ルビと傍点には上記のような詳細がある。しかしそれらを考慮すると、以下のような問題が起きる。

* ルビ文字に傍点をつけると同じ方向に二段必要になってしまう（両面ルビより幅広になってしまいやすい。画面占有率が無駄に高すぎる）し、傍点が小さすぎて強調されにくい
* 範囲に毎回開始と終了をつけると冗長（よって終了位置のメタ文字は省略可能とする。つまり開始位置だけにあり終了位置になければ最後の文字までがemとする）

このことから、傍点は親文字にのみ付与可能。ルビには傍点を付与できないものとする。もし`｜漢字《＊かんじ》`のように表記しても、ルビ文字の一文字目として`＊`が使われるだけだ。

#### 親文字に傍点をつける

```markdown
｜＊漢字《かんじ》
```
```html
<ruby class="under"><em>漢字</em><rt>かんじ</rt></ruby>
```

em併用rubyは必ず下ルビになる。emは強調という特性上、傍点が最初に目に入るべきだから。ルビは優先度が下がり下ルビになる。

`＊`で囲うと終端位置を示せる。

```markdown
｜＊漢＊字《かんじ》
```
```html
<ruby class="under"><em>漢</em>字<rt>かんじ</rt></ruby>
```

基本的には全親文字が対象とすれば、以下のように短縮表記できるはず。

```markdown
＊漢字《かんじ》
```

### 組合せ

両面＋分割ルビの組合せだけ可能である。em併用できないのは、上下段のいずれかが二段以上になるから。

両面|分割|em併用|可
----|----|------|--
❌|⭕|⭕|⭕
⭕|❌|⭕|❌
⭕|⭕|❌|⭕
⭕|⭕|⭕|❌

### 分割＋em併用ルビ

#### 親全体em

```markdown
＊新垣、結衣《あらがき、ゆい》
```
```html
<ruby class="under"><em>新垣</em><rt>あらがき</rt><em>結衣</em><rt>ゆい</rt></ruby>
```

#### 親一部em

```markdown
｜新＊垣、結＊衣《あらがき、ゆい》
```
```html
<ruby class="under">新<em>垣</em><rt>あらがき</rt><em>結</em>衣<rt>ゆい</rt></ruby>
```

`＊`の範囲指定は`、`の境界を超越可能。

残念ながら指定範囲は一箇所のみ。複数ヶ所の開始と終了を設定することはできない。複数可能にすると複雑になりすぎて「簡易記法」の守備範囲を超えるから。よって`＊`が三つ以上あれば不正。

#### 両面＋分割ルビ

両面ルビ＋分割ルビ。以下のような要件がある。氏名で分割しつつ、両面ルビを使いたい場合がある。ポイントは親文字のほうにも区切り文字が必要なことと、ルビ文字の片方にだけ同数の読点が必要なことだ。

```markdown
｜新垣、結衣《あらがき、ゆい｜ガッキー》
```
```html
<ruby class="under">
  <ruby class="over">新垣<rt>あらがき</rt></ruby>
  <ruby class="over">結衣<rt>ゆい</rt></ruby>
  <rt>ガッキー</rt>
</ruby>
```

もし親文字に非漢字があり、ルビ文字に読点があれば、非漢字の親文字は読点の機能も併せ持つ。つまり熟語ルビになる。

```markdown
｜伸縮自在の愛《しんしゅくじざい、あい｜バンジーガム》
```
```html
<ruby class="under">
  <ruby class="over">伸縮自在<rt>しんしゅくじざい</rt></ruby>
  の
  <ruby class="over">愛<rt>あい</rt></ruby>
  <rt>バンジーガム</rt>
</ruby>
```

この時、親文字の非漢字`の`だけは`<ruby>`文字がネストせず、テキストノードである。しかしそれは`<rt>バンジーガム</rt>`の親文字の一字である。

両面＋分割ルビが実現する条件は以下。違えば例外発生か読点をルビ文字にして警告表示する？

* 親文字とルビ文字の読点数は同じであるべき
* 読点は片方のルビ文字のみ許容する

##### 禁止：両ルビ読点

もし上下両方ルビ文字に同数の読点があれば、二つのルビ要素に分割すべきだ。すなわち以下のようにすべきだ。

```markdown
｜伸縮自在の愛《しんしゅくじざい、あい｜バンジー、ガム》
```
```html
<ruby>
  <ruby class="over">伸縮自在<rt>しんしゅくじざい</rt></ruby>
  <rt>バンジー</rt>
</ruby>
の
<ruby class="under">
  <ruby class="over">愛<rt>あい</rt></ruby>
  <rt>ガム</rt>
</ruby>
```

だがそれは以下のように表記したのと同じだ。

```markdown
伸縮自在《しんしゅくじざい｜バンジー》の愛《あい｜ガム》
```

本来「バンジーガム」という義訓は「伸縮自在の愛」という全部の語が揃って一つだからグループルビにすべきであり、分割して熟語ルビにすべきではない。

これらのことから、二つのルビ要素に分割する記法は作るべきでないと考えられる。

仮にグループルビにすべきルビ文字が上下共に存在しないなら、それらはそれぞれ二つの別々の両面ルビとして表記すべきである。それを略記すべく上下両方ルビに読点を含める記法を用意すると、グループルビがある場合との区別がつきにくくなる。それはルビの意味が読み取りにくくなってしまう。よって実装しないものとする。

つまり親文字に非漢字があればその字を読点位置と仮定するも、ルビ文字に読点がなければグループルビになるし、同数の読点があれば熟語ルビになる。但し両面ルビの場合は両方にルビがあると不正であり、片方にのみ読点がある形のみ許容すべきだ。

#### 禁止：両面＋em併用ルビ

両面＋em併用ルビは禁止だ。

理由は上段か下段が二段になってしまい画面占有率が高まりすぎることと、傍点の対象をルビ文字にすべきになり、その影響で傍点が小さくなり強調が弱まるから。かといって親文字に傍点を当てるとルビ文字と字数が合わないため奇妙な見え方になってしまい不適切だから。

以下のように両面ルビであれば、`＊`があってもem併用ルビにはならないこと。

```markdown
｜＊漢字《うえ｜した》
```
```markdown
｜漢字《＊うえ｜した》
```
```markdown
｜漢字《＊うえ｜した》
```
```markdown
｜漢字《うえ｜＊した》
```
```markdown
｜漢字《＊うえ｜＊した》
```
```markdown
｜＊漢字《＊うえ｜＊した》
```

### 禁止：両面＋分割＋em併用ルビ

両面とem併用は二段になるため禁止。
